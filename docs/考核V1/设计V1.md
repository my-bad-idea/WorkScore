### 6. 系统配置（system_config）
| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| score_weight_ai | AI考核权重 | 30% |
| score_weight_self | 自评权重 | 30% |
| score_weight_manual | 人工考核权重 | 40% |
| manual_score_min_count | 最低人工打分人数 | 3人 |

## 二、核心计算公式

```
单项最终占比    = (创建人占比 + 执行人设置占比) / 2

人工均值        = 启用去极值 ? 去掉最高分和最低分后取均值
                             : 所有人工打分直接取均值

单项综合得分    = score_ai          × weight_ai      （AI打分失败时按剩余权重重新归一化）
              + score_self        × weight_self     （自评缺失时按配置策略处理）
              + manual_avg        × weight_manual

月度总分        = Σ (单项综合得分 × 单项最终占比)
```

## 三、状态流转

```
待考核 ──[考核人申请考核]──▶ 考核中 ──[打分完成 + 负责人确认]──▶ 已考核
  │             │                │
  │         前置校验            打分截止后自动关闭打分入口
  │       ① 所有明细项执行人占比已填写
  │       ② 所有明细项占比总和 = 100%
  │       ③ 当月无周报时给出警示提醒
  │
  可自由编辑考核明细
```

## 四、权限规则

| 操作 | 可操作人 | 限制条件 |
|------|---------|---------|
| 新增/编辑考核明细 | 部门任意成员 | 仅限「待考核」状态 |
| 设置执行人占比 | 考核人本人 | 仅限「待考核」状态 |
| 填写完成情况 | 考核人本人 | 仅限「待考核」状态 |
| 申请考核 | 考核人本人 | 前置校验全部通过 |
| 自评打分 | 考核人本人 | 「考核中」且未超截止时间，每项限一次 |
| AI打分 | 系统自动触发 | 申请考核后自动执行，失败可由负责人重试 |
| 人工打分 | 部门任意成员 | 「考核中」且未超截止时间，每人每项限一次 |
| 确认完成考核 | 部门负责人 | 人工打分人数 ≥ 配置阈值，三方打分均已完成或截止 |
| AI打分重试/标记无效 | 部门负责人 | AI打分状态为「打分失败」时 |

## 五、打分依据说明

### AI 自动打分
申请考核后系统自动触发，输入以下信息生成分数，同步写入 `score_ai_remark` 供复核：

| 依据 | 说明 |
|------|------|
| 完成情况 | completion_status 枚举值 + completion_summary 概要文本 |
| 周报记录 | 考核月份内考核人全部周报内容，与考核项语义匹配评估执行程度 |
| 无周报情况 | 标注"无周报参考，仅依据完成情况打分"，分数仅供参考 |

**AI打分失败处理：**
```
打分失败 → score_ai_status = 打分失败
         → 负责人可选择：重试 或 标记无效
         → 标记无效后：weight_manual 和 weight_self 按比例归一化重新计算
```

---

### 人工打分
系统在打分界面同步展示以下参考信息，由成员手动录入分数：

| 参考信息 | 说明 |
|----------|------|
| 完成情况 | 考核明细项的完成情况枚举及概要说明 |
| 周报记录 | 考核月份内考核人的周报摘要 |
| AI打分说明 | score_ai_remark 内容，辅助人工参考 |

---

## 六、考核流程

```
① 部门成员为考核人录入考核明细
        ↓
② 考核人填写执行人占比、完成情况与概要
        ↓
③ 考核人申请考核
   ├─ 校验：所有明细项执行人占比已填写
   ├─ 校验：所有明细项占比之和 = 100%
   └─ 提示：当月无周报时给出警示（不阻断）
        ↓
   状态 → 「考核中」，周报与明细全部锁定，快照当前权重配置
        ↓
④ 三方同步进行打分（截止时间前）：
   ├─ 【自评】  考核人根据完成情况手动打分（每项限一次）
   ├─ 【AI】    系统自动读取完成情况 + 周报计算打分，失败时记录异常等待处理
   └─ 【人工】  部门成员参考完成情况 + 周报 + AI说明手动打分（每人每项限一次）
        ↓
   打分截止 → 自动关闭所有打分入口
        ↓
⑤ 负责人处理异常（如有）：
   ├─ AI打分失败 → 重试或标记无效
   └─ 人工打分人数不足 → 可配置是否允许继续或等待补打
        ↓
⑥ 部门负责人审核确认，填写审核意见
        ↓
⑦ 状态 → 「已考核」，系统按快照权重计算月度最终得分
```

# LLM评分一致性保证方案
这是一个很经典的LLM确定性问题。核心矛盾在于LLM本身是概率模型，天然存在随机性。以下是保证评分一致性的几个关键策略：

## 1. 控制模型参数

最直接的手段：

- **temperature = 0**：关闭随机采样，让模型每次选择概率最高的token
- **top_p = 1, top_k = 1**（如支持）：进一步压缩采样空间
- **固定seed**：部分模型（如OpenAI）支持 `seed` 参数，配合temperature=0基本可实现复现

```typescript
const response = await openai.chat.completions.create({
  model: "gpt-4o",
  temperature: 0,
  seed: 42,
  messages: [...]
});
```

但要注意：即使temperature=0，模型版本升级、后端负载均衡切换节点，仍可能导致结果漂移。

---

## 2. 结构化输出 + 评分量表

让模型输出**结构化JSON**而非自由文本，减少歧义空间：

```typescript
// Prompt中明确要求
const systemPrompt = `
你是一个周报评审专家，请严格按照以下量表对周报打分，输出JSON格式。

评分维度：
- 工作完成度（0-30分）：是否清晰列出本周完成事项
- 问题与风险（0-20分）：是否识别并说明了问题
- 下周计划（0-30分）：计划是否具体可执行
- 表达质量（0-20分）：内容是否简洁准确

评分规则：
- 每个维度按"优秀/良好/一般/不足"映射到对应分数区间
- 必须给出每项扣分理由

输出格式：
{
  "completion": { "score": number, "reason": string },
  "risk": { "score": number, "reason": string },
  "plan": { "score": number, "reason": string },
  "quality": { "score": number, "reason": string },
  "total": number,
  "summary": string
}
`;
```

量表越细化（用具体标准描述每个分段），模型的判断路径越确定，漂移越小。

---

## 3. Few-shot 示例锚定

在Prompt中放入**标准参考案例**，让模型有明确的对标基准：

```
【示例一】
周报内容：本周完成了XX模块开发，解决了YY问题，下周计划...
评分结果：{ "completion": 28, "reason": "事项具体，有产出描述" ... }

【示例二】
周报内容：本周做了一些开发工作。
评分结果：{ "completion": 10, "reason": "描述过于笼统，无具体产出" ... }
```

Few-shot示例是天然的"校准锚"，能显著降低评分分布的方差。

---

## 4. 多次采样取中位数（对抗残余随机性）

即使temperature=0，极端情况下仍可能出现波动。可以对同一份周报调用**3次**，取中位分数：

```typescript
async function scoreWithConsensus(report: string): Promise<number> {
  const scores = await Promise.all([
    scoreReport(report),
    scoreReport(report),
    scoreReport(report)
  ]);
  // 取中位数
  return scores.sort((a, b) => a - b)[1];
}
```

---

## 5. 结果缓存（兜底方案）

对相同输入做哈希缓存，直接复用历史结果，彻底规避模型随机性：

```typescript
import { createHash } from 'crypto';

const cache = new Map<string, ScoreResult>();

function getCacheKey(report: string, promptVersion: string): string {
  return createHash('sha256')
    .update(report + promptVersion)
    .digest('hex');
}

async function scoreWithCache(report: string): Promise<ScoreResult> {
  const key = getCacheKey(report, PROMPT_VERSION);
  if (cache.has(key)) return cache.get(key)!;
  
  const result = await callLLM(report);
  cache.set(key, result);
  return result;
}
```

注意缓存要跟**Prompt版本**绑定，Prompt一旦修改，旧缓存应失效。

## 综合建议

| 措施 | 一致性提升 | 成本 |
|------|-----------|------|
| temperature=0 + seed | ★★★★☆ | 零成本 |
| 结构化输出+细化量表 | ★★★★★ | 需设计 |
| Few-shot示例 | ★★★★☆ | 需准备 |
| 多次采样取中位 | ★★★☆☆ | 3倍费用 |
| 哈希缓存 | ★★★★★ | 存储成本 |

**实际推荐做法**：`temperature=0` + 精细化评分量表 + Few-shot示例，三者叠加后一致性已经非常高。若对准确性有极高要求，再叠加缓存兜底。
