# 需求&管理岗位考核标准（基于软件部周报汇总）- 评分一致优化版

适用对象：产品/需求分析（PRD、原型、验收标准）、项目管理/交付管理（计划、里程碑、过程跟踪、跨部门协调）、兼顾材料交付（操作手册/软著/审计支撑等）的岗位。

---

## 【评分输出约定】必读

- 评分时必须且仅能输出以下 **5 个考核项**，不得增删、不得改名，顺序可任意：
  1. **需求清晰与可追溯**
  2. **计划与过程推进**
  3. **验收与交付材料**
  4. **协同与支撑**
  5. **工作类型适配**
- 每项 0–100 分，按下方「各考核项评分区间」严格对照后给分；**必须**在 comment 中写清扣分理由或亮点（一句话即可）。
- 若周报中**未提及**某维度（如无任何计划/验收相关描述），该项按「无描述」档给分，并在 comment 中注明「周报未涉及」。
- **以证据为准**：需求池 ID、PRD/原型编号、里程碑、验收记录、材料清单等可追溯；无证据时对应维度不得打高分。

---

## 各考核项评分区间（严格按此对照）

### 1. 需求清晰与可追溯

| 分数区间 | 等级 | 判定条件（满足即落入该档） |
|----------|------|----------------------------|
| 85–100   | 优秀 | 有需求池条目（ID/优先级/状态/负责人等）或 PRD/原型/验收标准编号；需求-任务-版本可串起来；修订记录或变更点可查。 |
| 70–84    | 良好 | 有需求范围、PRD/原型或验收标准其中至少两项；有部分 ID 或版本对应关系；略缺完整追溯链。 |
| 55–69    | 一般 | 仅泛泛提到“写需求”“做原型”“澄清需求”等，无编号、无范围边界、无修订记录。 |
| 0–54     | 不足 | 几乎无需求/设计相关描述；或仅“参与需求”“协助沟通”等无法验收的表述。 |

- **无描述**：周报完全未提需求/PRD/原型/验收标准/需求池 → 给 0–30 分，comment 写「周报未涉及需求清晰与可追溯」。

### 2. 计划与过程推进

| 分数区间 | 等级 | 判定条件 |
|----------|------|----------|
| 85–100   | 优秀 | 有迭代计划、里程碑或任务拆分；有周度进度/节点达成率或阻塞清单；有变更评审或计划调整记录，可追溯。 |
| 70–84    | 良好 | 有计划/里程碑/进度/阻塞其中至少两项；有简单状态或达成说明；略缺量化或闭环记录。 |
| 55–69    | 一般 | 仅“跟进进度”“协调推进”等笼统描述，无计划节点、无里程碑、无阻塞清单。 |
| 0–54     | 不足 | 无计划或过程管理相关描述；或无法判断是否有效推进。 |

- **无描述**：未提计划/里程碑/进度/阻塞/风险 → 给 0–30 分，comment 写「周报未涉及计划与过程推进」。

### 3. 验收与交付材料

| 分数区间 | 等级 | 判定条件 |
|----------|------|----------|
| 85–100   | 优秀 | 有验收记录、一次通过率或上线公告；有操作手册/测试报告/软著/审计材料等清单或提交记录；上线问题有归因与复盘。 |
| 70–84    | 良好 | 有验收结论、上线或交付材料其中至少两项；有清单或完成度说明；略缺量化。 |
| 55–69    | 一般 | 仅“参与验收”“整理材料”等笼统描述，无记录、无清单、无版本/编号。 |
| 0–54     | 不足 | 无验收/上线/交付材料相关描述。 |

- **上线问题**：若提到上线后 P0/P1 问题且未说明归因与改进 → 该项不超过 75 分，comment 中须提及。
- **无描述**：未提验收/上线/材料交付 → 给 0–30 分，comment 写「周报未涉及验收与交付材料」。

### 4. 协同与支撑

| 分数区间 | 等级 | 判定条件 |
|----------|------|----------|
| 85–100   | 优秀 | 有跨部门/跨角色协调记录；有阻塞响应或闭环时长、会议纪要或对齐结论；有根因分析与预防措施或需求池维护记录。 |
| 70–84    | 良好 | 提到协调、阻塞清理、会议对齐、需求池维护或根因复盘其中至少 2 项；有简要结果或时效。 |
| 55–69    | 一般 | 仅“协调沟通”“开会”等笼统表述，无记录、无闭环结论。 |
| 0–54     | 不足 | 无协同或支撑相关描述。 |

- **无描述**：未提协调/阻塞/会议/需求池/复盘 → 给 0–30 分，comment 写「周报未涉及协同与支撑」。

### 5. 工作类型适配

| 分数区间 | 等级 | 判定条件 |
|----------|------|----------|
| 85–100   | 优秀 | 能识别主工作类型（需求澄清/原型文档/计划推进/验收材料/支撑协调/规划优化）且描述与证据匹配；有阶段或占比说明。 |
| 70–84    | 良好 | 工作类型清晰，内容与类型基本匹配，略缺占比或阶段说明。 |
| 55–69    | 一般 | 有工作内容但类型不清晰或与证据不匹配。 |
| 0–54     | 不足 | 无法判断工作类型；或内容过空无法适配。 |

- **无描述**：周报过短或过空无法判断 → 给 0–30 分，comment 写「周报内容过少无法判断类型」。

## 证据与数据来源（供对照周报内容）

评分时从周报中识别以下“证据字段”（满足越多、描述越具体，对应维度分数越高）：

- **需求池**：ID/优先级/状态/负责人/预计上线/变更记录
- **PRD 与设计**：需求说明书、原型、接口字段清单、验收标准（可链接或编号）
- **计划与里程碑**：迭代计划、任务拆分、节点对齐、周度进度、阻塞与风险清单
- **验收与上线**：验收记录、上线公告、变更说明、一次通过率
- **交付材料**：操作手册、测试报告、软著/审计材料清单与提交记录
- **协同**：协调纪要、答疑记录、会议结论、根因与预防措施、需求池维护

**统一原则**：以“需求池记录、PRD/原型、修订记录、里程碑、验收与交付材料”为证据；无 ID/无编号/无清单时对应维度不得打高分（优秀档需有至少 2 类证据或 3 项以上可追溯项）。

## 阶段/工作类型与权重参考（供综合判断）

- **A 需求对接与澄清**：需求拆分与优先级、验收标准完备率、答疑效率
- **B 原型与文档**：文档质量、修订与版本管理、原型覆盖
- **C 计划与过程管理**：里程碑推进、阻塞清理、跨角色对齐
- **D 验收上线与交付材料**：验收一次通过率、交付材料齐备率、上线问题控制
- **E 系统问题支撑与根因分析**：协调响应、根因与改进、需求池维护
- **F 规划与体验优化**：体验改进落地、产品规划输出

不同时间段可侧重不同类型（需求集中期重 A+B、交付期重 C、上线期重 D、支撑期重 E）；「工作类型适配」项体现当期主类型及占比；若周报未区分类型，则按描述整体判断后给分。

---

## 结果输出建议（供被考核人改进周报）

- 本周期负责的需求池范围（ID 清单/状态变化）
- 关键产出物：PRD/原型/里程碑/材料（列编号与完成度）
- 推进效果：按期交付、阻塞清理、验收结论（可量化）
- 风险与复盘：Top 1 风险 + 已落地预防措施（含验证口径）
