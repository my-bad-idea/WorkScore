# 月度考核系统设计文档

## 一、设计原则

本方案在系统化流程基础上融入客观考核机制，核心遵循三项原则：

**证据优先**：所有打分必须有可追溯的事实依据，禁止仅凭印象打分。

**多视角交叉**：自评、AI、同行、管理者四方视角相互印证，单一视角不决定结果。

**过程透明**：考核标准月初锁定，打分依据全程留存，被考核人有正式异议权。

---

## 二、考核维度与权重

### 通用四维框架

| 维度 | 说明 | 评分依据来源 |
|------|------|-------------|
| 交付质量 | 完成度、正确性、稳定性 | 任务记录、Bug记录、代码评审 |
| 工作效率 | 预计工期 vs 实际工期、自主解决问题能力 | 任务看板、周报 |
| 协作贡献 | 代码评审参与、文档完善、响应他人问题质量 | 评审记录、会议纪要 |
| 成长迭代 | 主动学习、技术分享、复盘改进 | 周报、分享记录 |

### 角色差异化权重

| 维度 | 初级工程师 | 中级工程师 | 高级/架构师 | 技术负责人 |
|------|-----------|-----------|------------|-----------|
| 交付质量 | 50% | 40% | 35% | 25% |
| 工作效率 | 25% | 25% | 20% | 15% |
| 协作贡献 | 15% | 20% | 25% | 30% |
| 成长迭代 | 10% | 15% | 20% | 30% |

> 维度权重在系统配置中按角色分组配置，月初锁定后当月不可修改。

## 四、核心计算公式

```
单项最终占比     = (创建人占比 + 执行人占比) / 2

同行均分         = 启用去极值 ? 去掉最高分和最低分后均值 : 直接均值

单项综合得分     = score_ai       × weight_ai         （失败时按剩余权重归一化）
               + score_self     × weight_self        （缺失时按配置策略处理）
               + peer_avg       × weight_peer
               + manager_score  × weight_manager

维度得分         = Σ (单项综合得分 × 单项最终占比)    [维度内各项]

月度总分         = Σ (维度得分 × 维度权重)            [四个维度加权]
```

## 五、状态流转

```
待考核 ──[月初占比对齐会 + 行为锚定确认]──▶ 考核中
  │                ↓
  │         前置申请校验：
  │         ① 行为锚定描述已填写
  │         ② 执行人占比已设置
  │         ③ 维度内占比合计 = 100%
  │         ④ 关联证据已填写（evidence_refs）
  │         ⑤ 当月无周报时警示提醒
  │
考核中 ──[四方打分完成 + 跨组校准通过]──▶ 结果公示
  │
结果公示 ──[异议窗口3天]──▶ 异议中（有异议）──[负责人处理]──▶ 已考核
                        └──▶ 已考核（无异议或异议窗口关闭）
```

## 六、权限规则

| 操作 | 可操作人 | 限制条件 |
|------|---------|---------|
| 新增/编辑考核明细 | 部门任意成员 | 仅限「待考核」状态 |
| 填写行为锚定描述 | 创建人 | 申请考核前，锁定后不可修改 |
| 设置执行人占比 | 考核人本人 | 仅限「待考核」状态 |
| 填写完成情况与证据 | 考核人本人 | 仅限「待考核」状态 |
| 申请考核 | 考核人本人 | 前置校验全部通过 |
| 自评打分 | 考核人本人 | 「考核中」且未超截止，每项限一次 |
| AI打分 | 系统自动 | 申请后自动触发，失败可重试或标记无效 |
| 同行评价 | 部门任意成员 | 「考核中」且未超截止，每人每项限一次，**必填说明** |
| 管理者打分 | 直属管理者 | 「考核中」状态，**必填说明**，校准前可修改 |
| 跨组校准 | 跨组管理者 | 管理者打分完成后，校准通过方可进入公示 |
| 发起异议 | 考核人本人 | 公示期内，窗口关闭后不再受理 |
| 处理异议 | 部门负责人 | 「异议中」状态 |
| 确认考核 | 部门负责人 | 无异议或异议已处理 |

## 七、打分依据规范

### 行为锚定评分参考（以交付质量为例）

| 分数段 | 行为描述 |
|--------|---------|
| 90-100 | 交付物零缺陷，主动发现并修复范围外潜在问题，交付物可直接作为团队规范参考 |
| 75-89 | 按时交付，偶发小问题可自主修复，不影响整体进度 |
| 60-74 | 基本完成，存在需他人协助排查或轻微延期情况 |
| 60以下 | 多次延期或交付物质量影响他人工作，需专项跟进 |

> 其余三个维度在系统配置中同样维护对应锚定描述，打分界面直接展示供参考。

### AI 打分
系统读取考核内容、行为锚定、完成情况、关联证据及当月周报，综合生成分数并输出说明，标注匹配到的周报关键依据，供人工复核。

### 同行评价 / 管理者打分
界面同步展示：行为锚定描述、完成情况与概要、关联证据、AI打分说明、当月周报摘要，打分时**必须填写具体事件依据**，不接受仅提交数字。

## 八、考核流程

```
【月初】
① 部门成员为考核人录入考核明细，填写行为锚定描述
② 月初召开"占比对齐会"（≤15分钟），当面确认各项考核占比
③ 考核人填写执行人占比、完成情况、关联证据
        ↓
【申请】
④ 考核人提交自述材料（主要工作、挑战、自我评价）
⑤ 系统执行前置校验（行为锚定/占比/证据/周报），通过后申请考核
   状态→「考核中」，所有明细、周报、行为锚定全部锁定，同步快照权重
        ↓
【四方打分】（截止日期前并行进行）
   ├─ 【自评】    考核人对照行为锚定与完成情况打分
   ├─ 【AI】      系统读取证据链自动打分，生成说明文档
   ├─ 【同行】    部门成员参考锚定/证据/周报打分，必填事件依据（每人每项一次）
   └─ 【管理者】  直属管理者参考全部材料打分，必填事件依据
        ↓
【校准】
⑥ 跨组管理者召开校准会议，对比组间分布，修正明显偏差
   校准完成后管理者打分锁定
        ↓
【公示与异议】
⑦ 结果向被考核人公示，开放 3 天异议窗口
⑧ 被考核人可提交异议，附支撑证据；负责人在窗口内处理并给出书面意见
        ↓
【确认】
⑨ 异议处理完毕或窗口关闭，部门负责人确认
   状态→「已考核」，系统按快照权重输出月度最终得分
        ↓
【反馈】
⑩ 一对一反馈面谈：重点讨论成长路径，而非解释分数
```

## 九、防偏见机制汇总

| 偏见类型 | 应对机制 |
|---------|---------|
| 近因效应 | 周报全程留存，证据链覆盖整月，月初行为锚定锁定标准 |
| 光环效应 | 四维度独立打分，单维度表现不影响其他维度 |
| 关系偏见 | 同行评价匿名，必填事件依据，不接受无依据打分 |
| 标准不一 | 行为锚定评分描述统一，跨组校准对齐组间尺度 |
| 大锅饭 | 强制分布预期（优20%/良50%/待改进30%），校准时对比组间分布 |
| 闷头做事者吃亏 | 被考核人自述材料先行，保障表达渠道 |
| 结果无法申诉 | 正式异议机制，3天窗口，书面处理留档 |
